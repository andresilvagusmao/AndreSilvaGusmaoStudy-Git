Como os sistemas de controle de versão distribuídos lidam com a propagação de mudanças entre repositórios?|As mudanças são propagadas por meio de uma série de operações de pull e push entre os repositórios locais e remotos.
Por que uma equipe pode escolher usar um sistema de controle de versão centralizado?|Uma equipe pode escolher um VCS centralizado por sua configuração mais simples, controle de acesso e gerenciamento centralizado dos arquivos do projeto.
Qual é a principal diferença entre um sistema de controle de versão centralizado e um distribuído?|A principal diferença é que em um sistema centralizado, todos os desenvolvedores trabalham em um único repositório central, enquanto em um sistema distribuído, cada desenvolvedor possui uma cópia completa do repositório, incluindo todo o histórico de mudanças.
Como os sistemas de controle de versão distribuídos lidam com conflitos durante a mesclagem?|Os conflitos são resolvidos manualmente pelos desenvolvedores, que escolhem a versão correta do arquivo ou combinam as mudanças de ambos os branches.
O que é um repositório remoto em um sistema de controle de versão distribuído?|Um repositório remoto é uma cópia do projeto armazenada em um servidor externo, usada para compartilhar mudanças e colaborar com outros desenvolvedores.
Qual é a principal vantagem de um sistema de controle de versão distribuído em relação a um centralizado?|Os sistemas de controle de versão distribuídos permitem uma maior flexibilidade e eficiência no trabalho, já que os desenvolvedores podem trabalhar em suas próprias cópias locais do repositório, sem a necessidade de conexão constante com o repositório central.
Quais são algumas desvantagens dos sistemas de controle de versão centralizados?|As desvantagens dos sistemas de controle de versão centralizados incluem: dependência de uma conexão constante com o repositório central, maior risco de perda de dados devido a um único ponto de falha e desempenho mais lento em comparação com os distribuídos.
